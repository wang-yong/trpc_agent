# 通用 AI Agent 设计方案与工具调用系统规划

> 创建时间：2026-06-22  
> 状态：设计与规划中  
> 目标：基于 `trpc-agent-go` 框架，打造一个具备工业级稳定性、自主推理能力和安全屏障的通用 AI Agent 平台。

---

## 一、当前框架 Tool Calling 支持机制剖析

在 `trpc-agent-go` 框架层，目前提供了一种基于 Go 反射（Reflection）和结构化映射的工具调用机制。其核心实现链条与生命周期如下：

### 1. 工具声明与结构化映射
在框架中，工具通过接口 `tool.Tool` 抽象，并通过 `function.NewFunctionTool(fn, options...)` 注册：
*   **参数映射**：框架使用 Go 的反射机制，自动解析 Go 函数 `fn` 的输入参数结构体（例如 `calculator.Input`）。
*   **Schema 自动生成**：利用结构体字段的类型、`json` 标签和 `description` 标签，框架在运行时自动生成符合 OpenAI / Anthropic 规范的 JSON Schema（Tool Definition），并在建立连接时作为 `tools` 字段传递给 LLM。

### 2. 参数解析与反序列化
*   当 LLM 决定调用某个工具时，会在响应中返回 `tool_calls` 结构，包含 `id`、`name` 和一段 JSON 格式的参数字符串（Arguments）。
*   框架层通过反射获取该工具对应的输入结构体类型，实例化一个空结构体，并利用 Go 标准库（或高效的第三方 JSON 库）进行反序列化（`json.Unmarshal`），将模型生成的 JSON 字符串解析为强类型的 Go 结构体实例。

### 3. 调用生命周期 (Execution Lifecycle)
一个完整的工具调用生命周期在双向 ReAct 循环中流转：
1.  **LLM 触发**：LLM 生成 `ToolCall` 指令，框架在客户端接收到完整的流式/非流式响应。
2.  **调度执行**：框架的 `LLMAgent` 循环拦截该调用，根据 `name` 检索已注册的 `Tool`。
3.  **参数校验与实例化**：反序列化参数。如果参数类型不匹配，则触发参数校验失败流程。
4.  **业务函数执行**：在独立的 Goroutine 或当前上下文协程中，调用具体绑定的 Go 函数（例如 `Calculate(ctx, input)`），传入 `context.Context` 以支持链路追踪（OTel）与超时控制。
5.  **捕获输出与异常**：获取业务函数返回的 `Output` 结构体和 `error`。

### 4. 结果处理与 ReAct 闭环
*   **序列化**：框架将 `Output` 结构体序列化为 JSON 字符串。
*   **上下文注入**：将调用结果封装为 `ToolMessage`（带上对应的 `tool_call_id`），并 append 到当前的会话历史消息列表中（Message History）。
*   **多轮迭代**：Runner 重新携带包含工具返回结果的最新历史记录再次发起 LLM 调用，直到 LLM 判定任务已完成并输出最终回答，或达到设定的迭代上限 `MaxToolIterations`。

## 二、通用 AI Agent 六大核心维度方案规划

现有的 `trpc-agent-go` 框架层提供了优秀的单步工具调用基建，但要将其演进为支撑复杂工程、沙箱运行和安全控制的**通用 AI Agent 平台**，必须在以下六大核心维度进行系统性的增强与实现：

```
                    ┌─────────────────────────┐
                    │     用户目标 (Goal)     │
                    └────────────┬────────────┘
                                 │
                                 ▼
                    ┌─────────────────────────┐
                    │   意图识别与智能路由器   │
                    └────────────┬────────────┘
                                 │
         ┌───────────────────────┼───────────────────────┐
         ▼                       ▼                       ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Agent A (开发)  │     │  Agent B (数据)  │     │  Agent C (网络)  │
└────────┬────────┘     └────────┬────────┘     └────────┬────────┘
         │                       │                       │
         └───────────────────────┼───────────────────────┘
                                 │ (动态工具发现 / MCP Server)
                                 ▼
                    ┌─────────────────────────┐
                    │    并行执行调度控制引擎  │◄─── [ 容错重试 & 自愈 ]
                    │  (DAG 依赖、Goroutine)  │
                    └────────────┬────────────┘
                                 │
         ┌───────────────────────┴───────────────────────┐
         ▼                                               ▼
┌─────────────────────────────────┐             ┌─────────────────────────────────┐
│        沙箱隔离执行引擎         │             │       高危工具人工干预审批      │
│  (Docker / PTY 终端 / 安全沙箱)  │             │     (Human-in-the-Loop Gate)    │
└─────────────────────────────────┘             └─────────────────────────────────┘
```

### 1. 工具注册与动态发现机制 (Dynamic Discovery)
*   **核心目标**：解耦静态编译依赖，支持工具在运行时动态注册、加载、卸载与热更新。
*   **实现范围**：
    *   **动态插件加载 (Go Plugin/.so)**：利用 Go `plugin` 包，支持动态加载编译好的 `.so` 文件工具集。
    *   **远程 MCP 协议对接**：实现 Model Context Protocol 客户端，支持读取远程或本地 MCP Server 所暴露的工具列表，并实时转化为 `trpc-agent-go` 的工具定义。
    *   **配置文件驱动**：引入 `tools.toml`，配置 Agent 启动时应自动激活的工具类别，支持在线启用/禁用指定工具。
*   **交付标准**：
    *   在不重启 Go 后台服务的前提下，通过修改配置文件或连接新 MCP 服务，前端可秒级感知并更新可用工具列表。

### 2. 意图识别与智能路由 (Intent Routing)
*   **核心目标**：当注册工具过多（如 50+ 个）时，防止全部传入 LLM 造成上下文污染（Context Stuffing）和意图混淆，实现高精度的工具子集路由。
*   **实现范围**：
    *   **工具元数据向量索引 (RAG Tool Selector)**：对所有工具的 `description` 进行向量化，根据用户当前输入的语义，计算余弦相似度，动态筛选出最相关的 Top-N 个工具注入 LLM 上下文。
    *   **分层路由架构**：采用“主 Agent 识别意图 -> 路由到垂直领域 Sub-Agent -> 调用领域内专用工具”的分层路由。
    *   **硬路由规则 (Hard Rules)**：支持正规表达式、关键词等前置硬路由规则，直接命中高优先级工具。
*   **交付标准**：
    *   在海量工具场景下，工具筛选准确率（Recall）大于 95%，且在路由阶段的耗时控制在 100ms 以内（本地向量检索）。

### 3. 执行调度与并行处理 (Parallel Execution Scheduler)
*   **核心目标**：支持 LLM 一次性输出的多个独立 Tool Calls 并行执行，并能够解决工具间的依赖执行。
*   **实现范围**：
    *   **并发 Goroutine 控制**：设计并发执行调度器，支持通过限制并发数的 Worker Pool 并发执行无相互依赖的 `tool_calls`。
    *   **DAG 有向无环图依赖解析**：当工具间存在数据流依赖（例如工具 B 需要工具 A 的输出结果）时，调度器动态解析依赖图，并按拓扑排序逐步执行。
    *   **单工具级别超时熔断**：为每个工具调用分配独立的 `context.WithTimeout`，防止单个工具因网络阻塞挂起整个 Agent 循环。
*   **交付标准**：
    *   支持并行执行多路网络搜索或文件读取，整体耗时相较于串行调用减少 60% 以上；单工具超时（默认 10s）自动终止且不影响其他并行任务。

### 4. 上下文状态与记忆管理 (Memory Management)
*   **核心目标**：在多次工具迭代中保持短期的会话运行状态（Stateful Tooling），并结合向量库建立长期工具调用经验。
*   **实现范围**：
    *   **运行态 Session 保持**：为交互式工具（如 Bash 终端、文件流、数据库连接）建立全局 Session 注册表。下一次调用能够延续上一次的工作目录（`cd`）或事务。
    *   **上下文滑动窗口压缩**：在多次 ReAct 思考循环中，如果上下文长度逼近模型极限，自动将历史的 `Thought-Action-Observation` 链条利用轻量级模型进行摘要提炼，保留最核心的记忆。
    *   **向量化长期记忆 (SQLite-VSS / pgvector)**：将成功的复杂任务工具执行链序列化并存储至本地数据库，在遇到类似复杂目标时，作为 Few-Shot Context 注入，引导 Agent 执行。
*   **交付标准**：
    *   Agent 可以在终端工具（Shell Tool）中跨多轮对话连续执行状态相关的指令（如先 `cd` 切换目录，后 `pwd` 或 `git status`，输出路径一致）。

### 5. 异常捕获与容错自愈 (Fault Tolerance & Self-Correction)
*   **核心目标**：当工具执行出错、参数格式错误或超时，Agent 能够自主识别并进行重试和自我纠错。
*   **实现范围**：
    *   **退避重试机制 (Exponential Backoff)**：对于因网络波动失败的 HTTP、搜索类工具，提供自动重试机制。
    *   **LLM 自愈闭环 (Self-Debugging Cycle)**：若工具内部返回业务错误，不直接中断任务，而是将错误堆栈或友好错误信息重新作为 `Observation` 返回给 LLM，允许 LLM 分析错误原因、修改参数并重新发起调用。
    *   **恐慌防御 (Panic Recovery)**：在调用第三方不稳定 `.so` 或代码段工具时，在外层包裹 `recover()`，防止单个工具崩溃导致整个服务器程序挂掉。
*   **交付标准**：
    *   任何工具抛出 Panic 均能被系统安全拦截，并记录审计日志；当由于参数缺失报错时，LLM 能够自动调整参数并在第二轮调用中成功执行。

### 6. 权限控制与安全隔离 (Security Sandbox)
*   **核心目标**：保护宿主机安全，防御恶意命令执行、敏感文件越权删除及超额 API 费用消耗。
*   **实现范围**：
    *   **高危动作人工确认门禁 (Human-in-the-Loop Gate)**：在调用涉及写操作（如写文件、删除文件、发送邮件、修改配置）的工具前，挂起执行流，向前端推送确认请求，只有用户点击“批准”后才允许继续。
    *   **轻量级进程沙箱隔离 (Sandbox Runtime)**：对于代码执行工具（Code Interpreter）或 Shell 终端工具，必须在受限环境（如 Docker 容器、gVisor、或者限制只读权限的工作目录）中拉起，严禁直接在宿主机运行。
    *   **频次与费用配额控制**：针对单个会话设置 Token 阈值和工具调用次数硬上限，防止死循环导致 API 账单暴涨。
*   **交付标准**：
    *   用户高危操作（如删除本地文件）必须百分百触发前端审批确认弹窗；代码执行工具与物理宿主机的敏感系统资源实现绝对隔离。

## 三、通用 Agent 设计落地与交付里程碑规划

为保证该通用 Agent 系统有序、高效落地，整体开发计划将划分为四个里程碑阶段：

### 🎯 里程碑 1：动态工具与 ReAct 自愈引擎 (M1)
*   **核心目标**：实现可自愈的 ReAct 思考链内核及动态工具热插拔。
*   **实现范围**：
    *   支持基于反射的运行时动态工具注册注册表。
    *   实现带 panic recover 与超时控制的生命周期管理器。
    *   在 HTTP SSE 流式推送中增加 `thought`（思考）事件，向前端传递思考链数据。
*   **交付标准**：后端可不重启注册新工具，前端能够看到实时渲染的 `Thought -> Action -> Observation` 步骤卡片，工具出错时 LLM 能自主修正参数再次调用。

### 🎯 里里程碑 2：高并发执行调度器与多 Session 保持 (M2)
*   **核心目标**：攻克高并发、多工具协同下的状态管理及执行效率瓶颈。
*   **实现范围**：
    *   构建支持 Goroutine Pool 的并行工具执行器。
    *   设计可保持 Session 的 Bash 终端和只读限制文件工具。
    *   支持简单的拓扑依赖工具链执行调度。
*   **交付标准**：可以同时在 3 个不同目录下并发检索文件，多轮会话内终端目录状态保持不丢失。

### 🎯 里程碑 3：RAG 智能路由与滑动上下文记忆 (M3)
*   **核心目标**：解决海量工具场景下的上下文爆满和长期记忆检索。
*   **实现范围**：
    *   引入本地轻量级向量存储（如 SQLite-VSS）。
    *   实现工具元数据 Embedding 生成，以及检索（RAG Tool Selector）。
    *   实现滑动窗口上下文压缩，对长历史进行定期自动摘要。
*   **交付标准**：注册 50 个工具，LLM 只接收到语义计算后最相关的 5 个工具，多轮极长对话下上下文占用不超过 8k tokens。

### 🎯 里程碑 4：安全沙箱、HITL 门禁及生态互联 (M4)
*   **核心目标**：筑牢安全防线，引入人工审批，并与标准 MCP 生态打通。
*   **实现范围**：
    *   设计 SSE 暂停与唤醒通道，实现“写动作”人工审批门禁（HITL）。
    *   将代码运行工具容器化，限制其最高可用 CPU、内存和只读目录。
    *   集成 Model Context Protocol，支持动态接入开源 MCP Server（如 Brave-Search 等）。
*   **交付标准**：执行写操作工具时，前端弹出醒目的审批对话框，用户批准后才能在受限沙箱中安全运行，并输出运行结果。
